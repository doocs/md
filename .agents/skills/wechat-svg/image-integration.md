# Image Integration And Pitfalls

## 1. `<image>` 的正确接入方式

核心原则：

- `<image>` 只是视觉内容，不应该自己决定时间轴
- 时间轴、旋转、进场和冻结全部由外层 `<g>` 接管
- `<image>` 必须属于某个对象模块，而不是脱链挂在 SVG 根部

标准结构：

```xml
<g> <!-- 遭遇物位移施加层 -->
  <animateTransform attributeName="transform" type="translate"
      begin="click" dur="80s"
      values="..."
      keyTimes="..."
      fill="freeze" restart="never" />

  <g> <!-- 遭遇物姿态层 -->
    <animateTransform attributeName="transform" type="rotate"
        begin="click" dur="80s"
        values="..."
        keyTimes="..."
        fill="freeze" restart="never" />

    <image x="0" y="-800" width="609" height="639" href="..." />
    <image x="0" y="-1500" width="609" height="634" href="..." />
    <image x="0" y="-1900" width="609" height="343" href="..." />
  </g>
</g>
```

## 2. `<image>` 的固定约束

- 不要依赖外部 CSS 调位置
- 不要把尺寸交给响应式布局
- 不要让 `<image>` 自己承担运动逻辑
- 不要把 `<image>` 放到与本体平级但不在同一局部姿态组的位置

推荐做法：

- 位置用 `x` / `y`
- 尺寸用 `width` / `height`
- 序列排布用不同的 `y`
- 组级运动用外层 `transform`

## 3. 文本卡与 `<image>` 的关系

如果对象既有文本卡，也有 `<image>`：

- 文本卡和 `<image>` 都应该属于同一对象的局部层级
- 但它们不一定要放在同一最小组里
- 判断标准不是「看起来近不近」，而是「是否应该共享同一轨迹和同一姿态」

## 4. 典型错误与修法

### 错误 1：`<image>` 放在 SVG 根部

- 结果：图片能显示，但完全脱离编组时间轴
- 修法：挂回对应遭遇物的姿态层或其内部子组

### 错误 2：`<image>` 跟在错误的补偿层下面

- 结果：图片方向对，但位置始终不对，或者跟着下一个对象一起移动
- 修法：确认它位于「当前对象的姿态层内部」，而不是「下一个对象的补偿层之后」

### 错误 3：只调 `x/y`，不看外层 `transform`

- 结果：怎么挪都不对，因为真正决定运动的是外层位移和旋转
- 修法：先定外层轨迹，再微调 `x/y`

### 错误 4：收尾字幕放在最外层

- 结果：视觉上居中，但逻辑上不属于最深层编组体系
- 修法：把字幕接入最深层补偿后的内部视口坐标层

### 错误 5：交互模式切换不彻底

- 结果：`<image>` 所在段落已经 click-only，但同一链上的遮挡层、字幕层或对象层仍保留旧回位逻辑
- 修法：统一整条链的 `begin` / `fill` / `restart`

## 5. 固定结论

### 结论 1

- `<image>` 是「局部视觉载荷」，不是「独立时间轴节点」

### 结论 2

- 居中文案虽然视觉上像 HUD，但如果它属于故事收尾的一部分，仍然应该放进最深层链内

### 结论 3

- 如果视觉感觉不对，优先排查时序、补偿层、父子链
- 不要第一反应只去改 `x/y/width/height`

## 6. 推荐排错顺序

1. 先看 `<image>` 在不在正确对象模块里
2. 再看它是不是放在姿态层里
3. 再看外层位移与旋转
4. 再看它的 `x/y/width/height`
5. 最后再看是否被透明度或遮挡层影响
